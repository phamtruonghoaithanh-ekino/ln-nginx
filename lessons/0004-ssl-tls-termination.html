Lesson 4: SSL/TLS Termination

Prerequisite: Lesson 2. Reference: Glossary.

The Big Idea

SSL/TLS termination means that nginx handles the encrypted HTTPS connection from the client, then talks plain HTTP to the backend server. The backend doesn't need to know anything about certificates — nginx takes care of it.

Client HTTPS (encrypted) nginx HTTP (plain) Backend ┌──────┐ ──────────────────────────▶ ┌──────────┐ ────────────────────────▶ ┌──────┐ │ │ ◀────────────────────────── │ TLS term │ ◀──────────────────────── │ │ └──────┘ └──────────┘ └──────┘ 🔒 🔓

This is called "termination" because the TLS connection ends (terminates) at nginx, and a new unencrypted connection starts to the backend.

Why terminate TLS at nginx?

  • Centralized config: One place to manage certs and TLS settings
  • Backend simplicity: App servers don't need to handle OpenSSL, cert rotation, etc.
  • Performance: nginx is highly optimized for TLS (session caching, OCSP stapling)
  • Flexibility: Route HTTP and HTTPS differently, redirect HTTP→HTTPS

Core Directives

server {
    listen              443 ssl;
    server_name         example.com;

    ssl_certificate     /etc/ssl/certs/example.com.pem;
    ssl_certificate_key /etc/ssl/private/example.com.key;

    # These are the modern defaults, but explicit is fine
    ssl_protocols       TLSv1.2 TLSv1.3;
    ssl_ciphers         HIGH:!aNULL:!MD5;

    # Performance: cache TLS sessions
    ssl_session_cache   shared:SSL:10m;
    ssl_session_timeout 10m;

    location / {
        proxy_pass http://localhost:3000;
    }
}
Directive What it does
listen 443 ssl; Enables SSL on this port (required, replaces old ssl on;)
ssl_certificate Path to the certificate file (PEM format, may include chain)
ssl_certificate_key Path to the private key (keep permissions 600!)
ssl_protocols Which TLS versions to allow. Modern: TLSv1.2 TLSv1.3
ssl_ciphers Which cipher suites to accept
ssl_session_cache Shared cache for TLS session resumption. shared:SSL:10m ≈ 40,000 sessions

HTTP → HTTPS Redirect

Almost every production site redirects HTTP to HTTPS:

server {
    listen 80;
    server_name example.com www.example.com;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl;
    server_name example.com www.example.com;

    ssl_certificate     /etc/ssl/certs/example.com.pem;
    ssl_certificate_key /etc/ssl/private/example.com.key;

    location / {
        proxy_pass http://localhost:3000;
    }
}

The first server block catches all HTTP traffic and permanently redirects to HTTPS. $host preserves the domain name; $request_uri preserves the path and query string.

Hands-on: Self-Signed Certificate

In production you'd use Let's Encrypt (free, automated) or a paid CA. For local testing, generate a self-signed cert:

Step 1: Generate cert and key

mkdir -p /tmp/ssl-demo/ssl

openssl req -x509 -newkey rsa:2048 -nodes \
  -keyout /tmp/ssl-demo/ssl/localhost.key \
  -out /tmp/ssl-demo/ssl/localhost.crt \
  -days 365 \
  -subj "/CN=localhost"

Step 2: nginx config with SSL + backend proxy

# /tmp/ssl-demo/nginx.conf

events {
    worker_connections 1024;
}

http {
    ssl_session_cache   shared:SSL:10m;
    ssl_session_timeout 10m;

    # HTTP → HTTPS redirect
    server {
        listen 80;
        server_name localhost;
        return 301 https://$host$request_uri;
    }

    # HTTPS server
    server {
        listen 443 ssl;
        server_name localhost;

        ssl_certificate     /tmp/ssl-demo/ssl/localhost.crt;
        ssl_certificate_key /tmp/ssl-demo/ssl/localhost.key;
        ssl_protocols       TLSv1.2 TLSv1.3;
        ssl_ciphers         HIGH:!aNULL:!MD5;

        location / {
            proxy_pass http://localhost:3000;
        }
    }
}

Step 3: Start a backend and test

# Simple backend
python3 -m http.server 3000 --bind 127.0.0.1 &

# Start nginx
nginx -c /tmp/ssl-demo/nginx.conf

# Test HTTP redirect (-L follows redirects, -k ignores self-signed)
curl -Lk http://localhost:80/
# → Should follow the redirect and show the directory listing from port 3000

# Test HTTPS directly
curl -k https://localhost:443/
# → Same content, but over TLS

Step 4: Verify TLS details

curl -kv https://localhost:443/ 2>&1 | grep -E '(SSL|TLS|cert|subject)'
# Shows: TLS handshake, certificate subject=/CN=localhost

Step 5: Clean up

nginx -s quit
kill %1
Why -k? Self-signed certs aren't trusted by your system. In production with a real CA, you never use -k — the cert is verified automatically. For testing locally, -k skips verification.

SSL Termination in Reverse Proxy

The real power: your backend app sees only HTTP, never touches TLS:

server {
    listen 443 ssl;
    server_name myapp.com;

    ssl_certificate     /etc/ssl/certs/myapp.com.pem;
    ssl_certificate_key /etc/ssl/private/myapp.com.key;

    # Add these so the backend knows the original protocol
    proxy_set_header X-Forwarded-Proto https;
    proxy_set_header X-Forwarded-Port   443;

    location / {
        proxy_pass http://localhost:3000;
    }
}

The backend receives the request as if it came from a normal HTTP client, except with X-Forwarded-Proto: https so it can generate correct absolute URLs (e.g., https://... instead of http://...).

Production Best Practices

Practice Why
Use Let's Encrypt / certbot Free, automated certs with auto-renewal
Only TLSv1.2 and TLSv1.3 Older versions (SSLv3, TLSv1.0/1.1) have known vulnerabilities
Set ssl_session_cache TLS handshakes are CPU-heavy; caching avoids them for repeat visitors
Redirect HTTP → HTTPS Never serve content over unencrypted connections
Restrict private key permissions chmod 600 /etc/ssl/private/*.key — only root can read
Note: nginx's default settings (ssl_protocols TLSv1.2 TLSv1.3 and ssl_ciphers HIGH:!aNULL:!MD5) are already quite secure. You don't need elaborate cipher configuration unless you have specific compatibility requirements.

What You Learned

  • SSL/TLS termination = nginx handles HTTPS, backend speaks HTTP
  • listen 443 ssl; enables TLS on a port
  • ssl_certificate + ssl_certificate_key point to cert and key files
  • return 301 https://$host$request_uri redirects HTTP→HTTPS
  • ssl_session_cache reduces TLS handshake overhead
  • Self-signed certs are fine for local dev; use Let's Encrypt for production
  • Forward X-Forwarded-Proto: https so backends know the original scheme
