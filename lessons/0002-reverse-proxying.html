Lesson 2: Reverse Proxying

Prerequisite: Lesson 1. Reference: Glossary.

The Big Idea

A reverse proxy sits in front of one or more backend servers and forwards client requests to them. The client talks only to nginx; the backend is never directly exposed. This is one of nginx's most common use cases.

┌──────┐ request ┌──────────┐ forward ┌──────────┐ │Client│ ──────────▶ │ nginx │ ──────────▶ │ Backend │ │ │ ◀────────── │ (proxy) │ ◀────────── │ (app) │ └──────┘ response └──────────┘ response └──────────┘

Why use a reverse proxy?

  • Hide internals: Backend apps (Node, Python, Ruby) listen on private ports; only nginx is public on 80/443.
  • SSL termination: nginx handles HTTPS; the backend speaks plain HTTP.
  • Load balancing: Distribute traffic across multiple backend instances.
  • Caching: Cache backend responses in nginx to reduce load.
  • Static offload: nginx serves static files (CSS, JS, images) directly — the backend never sees them.
  • Buffering: nginx buffers slow backend responses so clients receive data efficiently.

The Core Directive: proxy_pass

The simplest reverse proxy config. Everything starting with /api/ goes to a backend:

server {
    listen 80;

    location /api/ {
        proxy_pass http://localhost:3000;
    }

    location / {
        root /var/www/static;
    }
}

Requests to /api/users get forwarded to http://localhost:3000/api/users. Requests to /index.html are served from disk — the backend never sees them.

No trailing slash? If proxy_pass has a URI path (e.g. http://localhost:3000/ with trailing /), it replaces the matched location prefix. If it has no path (e.g. http://localhost:3000), the full original URI is passed through.
Config Request Backend receives
proxy_pass http://b:3000; /api/foo /api/foo
proxy_pass http://b:3000/; (with trailing /) /api/foo /foo (prefix replaced)

Forwarding Client Information

When nginx proxies a request, the backend sees nginx's IP, not the real client. Fix this by forwarding headers:

location /api/ {
    proxy_pass http://localhost:3000;

    proxy_set_header Host               $host;
    proxy_set_header X-Real-IP          $remote_addr;
    proxy_set_header X-Forwarded-For    $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto  $scheme;
}
Header What it contains
X-Real-IP The client's real IP ($remote_addr)
X-Forwarded-For Chain of IPs the request passed through
X-Forwarded-Proto http or https (so the backend knows if SSL was used)

Hands-on: Proxy a Python HTTP Server

Step 1: Start a simple Python backend

mkdir -p /tmp/proxy-demo/backend

cat << 'PYEOF' > /tmp/proxy-demo/backend/app.py
from http.server import HTTPServer, BaseHTTPRequestHandler

class Handler(BaseHTTPRequestHandler):
    def do_GET(self):
        body = (
            f"<h1>Hello from Python!</h1>\n"
            f"<pre>\n"
            f"Path: {self.path}\n"
            f"X-Real-IP: {self.headers.get('X-Real-IP', 'none')}\n"
            f"X-Forwarded-For: {self.headers.get('X-Forwarded-For', 'none')}\n"
            f"X-Forwarded-Proto: {self.headers.get('X-Forwarded-Proto', 'none')}\n"
            f"</pre>"
        ).encode()
        self.send_response(200)
        self.send_header('Content-Type', 'text/html')
        self.send_header('Content-Length', str(len(body)))
        self.end_headers()
        self.wfile.write(body)

HTTPServer(('', 3000), Handler).serve_forever()
PYEOF

python3 /tmp/proxy-demo/backend/app.py &

Step 2: nginx config that proxies to it

# /tmp/proxy-demo/nginx.conf

events {
    worker_connections 1024;
}

http {
    server {
        listen 8080;
        server_name localhost;

        # Proxy /api/ to the Python backend
        location /api/ {
            proxy_pass http://localhost:3000/;  # trailing / strips /api/

            proxy_set_header Host              $host;
            proxy_set_header X-Real-IP         $remote_addr;
            proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
        }

        # Serve static files directly (no backend)
        location / {
            root /tmp/proxy-demo/www;
        }
    }
}
mkdir -p /tmp/proxy-demo/www
echo '<h1>Static page (served by nginx directly)</h1>' > /tmp/proxy-demo/www/index.html

Step 3: Start nginx

nginx -c /tmp/proxy-demo/nginx.conf

Step 4: Test

# Static content — served by nginx, no backend involved
curl http://localhost:8080/
# → <h1>Static page (served by nginx directly)</h1>

# Proxied to Python backend
curl http://localhost:8080/api/test
# → <h1>Hello from Python!</h1>
#   Path: /test
#   X-Real-IP: 127.0.0.1
#   X-Forwarded-For: 127.0.0.1

Step 5: Clean up

nginx -s quit
kill %1  # stop the Python server
Key insight: The backend sees /test (not /api/test) because we used proxy_pass http://localhost:3000/; with a trailing slash, which replaces the /api/ prefix. Try removing the trailing / and see what changes.

Buffering (and When to Turn It Off)

By default, nginx buffers responses from the backend. This means:

  1. nginx reads the entire response from the backend into buffers.
  2. Then sends it to the client at network speed.

Benefits: frees up the backend quickly, handles slow clients efficiently.

Downside: adds latency for the first byte (TTFB). Turn it off for real-time apps:

location /api/ {
    proxy_buffering off;
    proxy_pass http://localhost:3000;
}

Timeouts

Backend connections can hang. nginx gives you safety nets:

location /api/ {
    proxy_connect_timeout   5s;   # how long to wait for the TCP handshake
    proxy_read_timeout      30s;  # how long between successive reads from backend
    proxy_send_timeout      30s;  # how long between successive writes to backend
    proxy_pass http://localhost:3000;
}

What You Learned

  • A reverse proxy forwards client requests to backend servers while hiding them
  • proxy_pass is the core directive for proxying
  • Trailing / on proxy_pass replaces the location prefix; no trailing slash passes the full URI
  • Forward X-Real-IP, X-Forwarded-For, X-Forwarded-Proto so backends know the real client
  • Buffering is on by default; disable with proxy_buffering off for real-time apps
  • Set timeouts to avoid hanging connections to unhealthy backends
