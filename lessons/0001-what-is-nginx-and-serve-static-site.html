Lesson 1: What Is nginx?

Part of the nginx learning track. See also: Glossary & Quick Reference

The Big Idea

nginx (pronounced "engine x") is a web server that can also act as a reverse proxy, load balancer, HTTP cache, and TLS terminator. It was created by Igor Sysoev in 2004 to solve the C10K problem — handling 10,000 simultaneous connections without crashing or consuming all your RAM.

Unlike older servers (Apache) that spawn a process or thread per connection, nginx uses an event-driven, asynchronous, non-blocking architecture. One worker process can handle thousands of connections at once.

How It's Different from Apache

Feature Apache nginx
Connection model Process/thread per connection Event-driven, single-threaded workers
Memory per connection ~1 MB+ ~550 bytes (idle keepalive)
Static files Good Excellent (sendfile + async I/O)
Dynamic content Built-in (mod_php, etc.) Proxy to app server (FastCGI, uwsgi, proxy_pass)
Config .htaccess per directory Single centralized config

Architecture in One Diagram

┌─────────────────────────────────────────────────────┐ │ nginx Master Process │ │ Reads config · binds sockets · spawns/reaps workers │ └────────┬──────────┬──────────┬──────────┬────────────┘ │ │ │ │ ┌────▼───┐ ┌────▼───┐ ┌────▼───┐ ┌────▼───┐ │ Worker │ │ Worker │ │ Worker │ │ Worker │ │ #1 │ │ #2 │ │ #3 │ │ #4 │ └───┬────┘ └───┬────┘ └───┬────┘ └───┬────┘ │ │ │ │ ┌───▼──────────▼──────────▼──────────▼───┐ │ Event loop (epoll/kqueue) per worker │ │ Handles thousands of connections │ │ Non-blocking I/O on all operations │ └──────────────────────────────────────────┘

Each worker runs an event loop (like Node.js). It accepts new connections, reads requests, processes them, and sends responses — all without blocking. When one request is waiting for disk or upstream, the worker handles other requests.

The Configuration File

nginx configuration lives in a single file (or files included from it), typically /etc/nginx/nginx.conf or /usr/local/etc/nginx/nginx.conf. The syntax is C-style — braces { }, semicolons ;, and hashes # for comments.

Directives and Contexts

A directive is a name + parameters + ;. A block directive opens a { } context. The major contexts are:

main # global settings ├── events # connection processing method └── http # HTTP server configuration ├── upstream # group of backend servers ├── server # virtual host (domain + port) │ ├── location # URL path matching rules │ └── location └── server └── location

Key rule: contexts never overlap. A location only lives inside a server, which lives inside http. No exceptions.

Storage tip: nginx does not support .htaccess files. All configuration is centralized. This is by design — it keeps things predictable and performant.

Hands-on: Serve Your First Static Site

Let's make nginx serve a simple HTML page from your local machine.

Step 1: Create a site directory

mkdir -p /tmp/nginx-demo/www
echo '<h1>Hello from nginx!</h1>' > /tmp/nginx-demo/www/index.html

Step 2: Create a minimal config

# /tmp/nginx-demo/nginx.conf

events {
    worker_connections  1024;
}

http {
    server {
        listen 8080;

        location / {
            root /tmp/nginx-demo/www;
        }
    }
}

Let's break this down:

  • events { } — required context. Sets connection processing.
  • http { } — wraps all HTTP-related config.
  • server { } — one virtual host, listening on port 8080.
  • location / { } — matches all URLs starting with /.
  • root ... — maps the URL to a filesystem path.

Step 3: Start nginx with your config

nginx -c /tmp/nginx-demo/nginx.conf
Note: You may need sudo depending on your system. On macOS with Homebrew, run sudo nginx -c /tmp/nginx-demo/nginx.conf.

Step 4: See it live

curl http://localhost:8080/
# → <h1>Hello from nginx!</h1>

Or open http://localhost:8080/ in a browser.

Step 5: Stop nginx when done

nginx -s quit

✏ Your Turn: Modify the Config

  1. Change the port from 8080 to 8081.
  2. Add a second location block that maps /data to /tmp/nginx-demo/data (create this directory and a file in it).
  3. Reload with nginx -s reload and test both paths.

How Request Matching Works

When a request arrives, nginx picks a location in this order:

  1. Prefix match: scan all location /path directives, remember the longest matching prefix.
  2. Regex match: scan all location ~ pattern directives. If any matches, use it instead of the remembered prefix.
  3. If no regex matches, use the remembered prefix.

Example from the config above:

location / {          # prefix, length 1 — catch-all
    root /tmp/nginx-demo/www;
}
location /images/ {  # prefix, length 8 — more specific
    root /tmp/nginx-demo;
}
location ~ \.(gif|jpg|png)$ {  # regex — overrides both
    root /tmp/nginx-demo/images;
}

A request to /images/cat.jpg would:

  1. Match both prefix locations — /images/ (longer) wins so far.
  2. Then match the regex \.(gif|jpg|png)$ — regex wins overall.
  3. File served from /tmp/nginx-demo/images/images/cat.jpg (root + URI).
Gotcha: root appends the URI to the path. So root /data; location /images/ { } serves /data/images/.... To strip the location prefix, use alias instead.

Controlling nginx

Signal Command What it does
stop nginx -s stop Immediate shutdown
quit nginx -s quit Graceful shutdown (finish current requests)
reload nginx -s reload Reload config without downtime
reopen nginx -s reopen Reopen log files (for log rotation)

How reload works: The master process checks the new config syntax, then starts new workers with the new config, and signals old workers to stop accepting connections and finish their current requests. Zero-downtime.

What You Learned

  • nginx is an event-driven web server that handles high concurrency efficiently
  • It has one master process + several worker processes
  • Config is centralized, C-style, with nested contexts (http → server → location)
  • Prefix locations are compared by length; regex locations override them
  • You served your first static site on port 8080
  • You can reload without downtime
