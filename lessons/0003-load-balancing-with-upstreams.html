Lesson 3: Load Balancing with Upstreams

Prerequisite: Lesson 2. Reference: Glossary.

The Big Idea

A single backend server is a single point of failure. If it goes down, your site goes down. If it gets overwhelmed, your site slows down. Load balancing spreads traffic across multiple backend servers for reliability and scalability.

┌──────────┐ ┌──▶│ Backend A│ │ └──────────┘ ┌──────┐ ┌──────┴──┐ ┌──────────┐ │Client│───▶│ nginx │────────────▶│ Backend B│ └──────┘ │upstream │ (round- └──────────┘ │ myapp │ robin) ┌──────────┐ └──────────┘──────────▶│ Backend C│ └──────────┘

nginx uses an upstream block to define a group of backend servers. You then reference this group by name in proxy_pass.

The upstream Block

An upstream lives inside http { }, at the same level as server blocks:

http {
    upstream myapp {
        server 127.0.0.1:3000;
        server 127.0.0.1:3001;
        server 127.0.0.1:3002;
    }

    server {
        listen 8080;

        location / {
            proxy_pass http://myapp;
            #          ______↑
        }
    }
}

nginx distributes requests across the three backends using the round-robin method by default (one request to each server in turn).

Load Balancing Methods

Method Directive When to use
Round-robin (default) Backends are similar and requests are uniform
Least connections least_conn; Requests vary in processing time; avoids overloading busy servers
IP hash ip_hash; Sticky sessions — same client always → same backend
Generic hash hash $request_uri; Cache affinity — same URL always → same backend
Random random two least_conn; Large upstream groups; avoids thundering herd on connection storms 
upstream myapp {
    # Pick ONE method (or omit for default round-robin)
    least_conn;
    # ip_hash;
    # hash $remote_addr consistent;
    # random two least_conn;

    server 127.0.0.1:3000;
    server 127.0.0.1:3001;
    server 127.0.0.1:3002;
}

Server Parameters

Each server in an upstream group accepts parameters:

upstream myapp {
    server 127.0.0.1:3000 weight=3;
    server 127.0.0.1:3001;
    server 127.0.0.1:3002             backup;
    server 127.0.0.1:3003             down;
}
Parameter Default What it does
weight= 1 Proportion of traffic this server receives. weight=3 gets 3x the requests.
max_fails= 1 Consecutive failures before marking server as unavailable.
fail_timeout= 10s 1) Time window for max_fails. 2) How long the server stays marked as down.
backup — Only receives traffic when all primary servers are down.
down — Permanently unavailable — useful for maintenance without removing from config.
How health checks work: After a server is marked as failed (max_fails failures in fail_timeout seconds), nginx stops sending it traffic. After fail_timeout passes, it probes with a live client request. If that succeeds, the server is back in rotation. This is passive health checking — it requires real traffic.

Hands-on: Load Balance Across 3 Python Servers

Step 1: Start 3 backends that each identify themselves

mkdir -p /tmp/lb-demo

for i in 3000 3001 3002; do
  cat << PYEOF > /tmp/lb-demo/server$i.py
from http.server import HTTPServer, BaseHTTPRequestHandler

class Handler(BaseHTTPRequestHandler):
    def do_GET(self):
        body = f"<h1>Server on port {self.server.server_port[1]}</h1>".encode()
        self.send_response(200)
        self.send_header('Content-Type', 'text/html')
        self.end_headers()
        self.wfile.write(body)

HTTPServer(('', $i), Handler).serve_forever()
PYEOF
  python3 /tmp/lb-demo/server$i.py &
done

Step 2: nginx config with upstream

# /tmp/lb-demo/nginx.conf

events {
    worker_connections 1024;
}

http {
    upstream backend {
        server 127.0.0.1:3000;
        server 127.0.0.1:3001;
        server 127.0.0.1:3002;
    }

    server {
        listen 8080;

        location / {
            proxy_pass http://backend;
        }
    }
}

Step 3: Start nginx and test

nginx -c /tmp/lb-demo/nginx.conf

# Run this 6 times — see the round-robin pattern
for i in 1 2 3 4 5 6; do
  curl -s http://localhost:8080/
done

# You should see alternating:
#   Server on port 3000
#   Server on port 3001
#   Server on port 3002
#   Server on port 3000
#   Server on port 3001
#   Server on port 3002

Step 4: Test passive health checks

# Kill one backend
kill %1  # kill the first one (port 3000)

# Requests will now only go to 3001 and 3002
for i in 1 2 3 4; do
  curl -s http://localhost:8080/
done

Step 5: Weighted round-robin

Edit the upstream block to give port 3000 more weight:

upstream backend {
    server 127.0.0.1:3000 weight=3;
    server 127.0.0.1:3001;
    server 127.0.0.1:3002;
}
nginx -s reload

# 3000 should get 3x more requests than 3001 or 3002
for i in 1 2 3 4 5; do
  curl -s http://localhost:8080/
done
# Expected: 3000, 3000, 3000, 3001, 3002

Step 6: Clean up

nginx -s quit
kill %1 %2 %3 2>/dev/null

When to Choose Each Method

Scenario Method Why
Stateless API servers round-robin Simple, fair, no session concerns
Variable request durations least_conn Avoids piling up on a server with long-lived requests
Sessions stored in memory ip_hash Same client always hits the same server
HTTP caching layer hash $uri consistent Same URL → same server → higher cache hit ratio
Very large upstream (10+) random two least_conn Picks two at random, selects the less loaded. Scales well.

What You Learned

  • upstream blocks group backend servers for load balancing
  • Default method is round-robin; least_conn, ip_hash, hash, random for other needs
  • Server parameters: weight, max_fails, fail_timeout, backup, down
  • Passive health checks: nginx detects failures and stops sending traffic automatically
  • Weights control traffic proportions (e.g. powerful servers get weight=3)
  • proxy_pass http://upstream_name references the group
